## Project Overview

GeroExplorer is an interactive graph visualization of aging-disease relationships.
Core data model: **Hallmarks of Aging → Mechanisms → Diseases** (all with confidence-scored edges).

See README.md for full project description.

---

## Architecture

### Data layer (`data/`)
Three JSON source files → assembled into `graph.json` by `scripts/build_graph.py`.

- `hallmarks.json` — 12 hallmark objects (Lopez-Otín 2013 + 2023)
- `mechanisms.json` — mechanism objects with `hallmark_links[]`
- `diseases.json` — disease objects with `mechanism_links[]` and `hallmark_direct_links[]`
- `graph.json` — D3-ready `{ nodes[], links[] }` — **never edit directly, always rebuild**
- `pubmed_cache.json` — generated by `fetch_pubmed.py`, gitignored
- `pubtator_raw.json` — raw PubTator API results, gitignored
- `pubtator_candidates.json` — proposed new diseases from PubTator (requires human review before merge)

### Data pipeline (`scripts/`)
- `build_graph.py` — assembles graph.json from data files; copies to `frontend/public/graph.json`
- `fetch_pubmed.py` — bibliometric confidence calibration via NCBI Entrez (0.35s/req rate limit)
- `fetch_pubtator.py` — disease discovery and confidence scoring via PubTator3 NER (see below)

### Frontend (`frontend/`)
React 18 + D3 v7 + Vite 5. No CSS framework — plain CSS with dark theme (`#0f172a`).

Key component: `ForceGraph.jsx` — D3 force simulation in SVG, `useRef`/`useEffect` integration.
Graph data is served from `frontend/public/graph.json` at `/graph.json`.

## Confidence Score Architecture (decided 2026-03-22)

Three confidence sources, each stored separately. Users can switch between them live in the frontend.

### Source 1: Bibliometric (default)
- **What:** Log-scaled PubMed publication count per search term
- **Script:** `fetch_pubmed.py`
- **Strength:** Fast, fully automatic, broad coverage
- **Weakness:** Counts co-mention, not causality. Biased toward well-studied diseases.
- **Cache key:** `"bibliometric"` in `pubmed_cache.json`

### Source 2: PubTator NER
- **What:** Co-occurrence of disease and hallmark entities across pre-tagged PubMed abstracts
- **Script:** `fetch_pubtator.py`
- **Strength:** Structured entity-level signal, covers entire PubMed, pre-processed by NCBI
- **Weakness:** Co-occurrence ≠ causality. Cannot distinguish direction or mechanism.
- **Cache key:** `"pubtator"` in `pubtator_cache.json`
- **Also used for:** Automated disease discovery (see below)

### Source 3: Abstract-parsing (planned)
- **What:** NLP over abstract text to extract relationship direction and linguistic certainty
- **Script:** not yet built
- **Strength:** Can distinguish "A causes B" from "A correlates with B"; extracts effect direction
- **Weakness:** Slower, requires NLP model, more error-prone
- **Cache key:** `"abstract_nlp"` in `pubmed_cache.json`
- **Combination:** PubTator NER as first filter (cheap, broad) → abstract-parsing on top-scoring edges only

### Combined confidence
When multiple sources are available, the frontend displays all three and lets the user select.
`build_graph.py` uses the source specified by `--confidence-source` flag (default: `bibliometric`).

## Disease Discovery Pipeline (decided 2026-03-22)

New diseases are discovered via PubTator NER, not written manually.

### Workflow
```
Phase 1 — Explore:   fetch_pubtator.py --phase explore
                     → Queries PubTator for each hallmark
                     → Prints distribution statistics
                     → Human reviews and decides threshold

Phase 2 — Generate:  fetch_pubtator.py --phase generate --min-count N --min-relative F
                     → Applies tiered threshold filter
                     → Writes data/pubtator_candidates.json (NOT diseases.json)
                     → Requires human review

Phase 3 — Merge:     fetch_pubtator.py --phase merge --confirm
                     → Merges approved candidates into diseases.json
                     → Run build_graph.py after
```

### Tiered threshold filter
A disease is included if it passes **both** criteria:
```python
count >= min_absolute       # default: 20  — filters pure noise
AND
count / total >= min_relative  # default: 0.001 — normalizes for rare vs. common diseases
```
Rationale: a fixed absolute threshold is biased against rare diseases with small literature.
Hutchinson-Gilford Progeria with 15/400 papers is a stronger signal than Type 2 Diabetes with 15/200,000.

**Empirical basis (full pipeline run 2026-03-22):**
- 743 unique entities found across all 12 hallmarks (4800 PMIDs total)
- Distribution: median count = 2, p75 = 4–7 across all hallmarks
- Count ≥ 20: yields 3–14 per hallmark (~100–150 total) — chosen as first-pass threshold
- Count ≥ 10: yields 13–28 per hallmark (~200–300 total) — available for broader sweep later

**Additional exclusion terms (beyond infectious/ICD-10 rules):**
Death, Drug-Related Side Effects, Insulin Resistance (mechanism), Chromosomal Instability (process)

### Exclusion rules
- **Infectious diseases**: excluded by default (keyword filter on disease name)
  - Keywords: infection, virus, bacterial, fungal, parasit, HIV, COVID, influenza, tuberculosis, malaria, hepatitis, pneumonia, sepsis
- **ICD-10 scope**: default mode only includes diseases likely to have ICD-10 codes
- **Syndromes**: excluded by default; include with `--include-syndromes` flag

## Planned Features (not yet built)

### Layout improvements (decided 2026-03-22)
- Highlight mode: selected node + 1-hop neighbors at full opacity; 2-hop at 60%; rest at 10%
- Dynamic spacing: D3 repulsion increases for highlighted subgraph on selection → nodes "breathe out"
- Labels: show on hallmarks and mechanisms by default; show on all highlighted nodes on selection
- Soft animated transition (low alpha reheat, not a hard reset)

### Disease overlap view (decided 2026-03-22)
- Clicking a disease shows "sibling diseases" — other diseases sharing the same mechanisms
- Grouped by shared mechanism (e.g. "via Endothelial Dysfunction: Atherosclerosis, Hypertension...")
- Scientifically: reveals mechanistic basis for known clinical comorbidities

## Important conventions

- IDs: snake_case prefixed by type — `hallmark_*`, `mech_*`, `disease_*`
- Node types in graph.json: `"hallmark"`, `"mechanism"`, `"disease"`
- Edge types: `"hallmark_to_mechanism"`, `"mechanism_to_disease"`, `"hallmark_to_disease"`
- Always rebuild after data changes: `python3 scripts/build_graph.py`
- `frontend/public/graph.json` is updated automatically on rebuild

## Development

```bash
# Rebuild graph after data changes
python3 scripts/build_graph.py

# Frontend dev server
cd frontend && npm run dev

# PubMed confidence calibration
python3 scripts/fetch_pubmed.py --email your@email.com --dry-run
python3 scripts/fetch_pubmed.py --email your@email.com

# PubTator disease discovery
python3 scripts/fetch_pubtator.py --phase explore              # see distribution
python3 scripts/fetch_pubtator.py --phase explore --limit 2    # test with 2 hallmarks
python3 scripts/fetch_pubtator.py --phase generate --min-count 20 --min-relative 0.001
python3 scripts/fetch_pubtator.py --phase merge --confirm
```

## What NOT to do

- Do not edit `data/graph.json` directly — it is generated
- Do not edit `data/pubtator_candidates.json` without reviewing it first — it requires human approval
- Do not hardcode disease/mechanism data into the frontend — always read from graph.json
- Do not add frontend frameworks (no Tailwind, no MUI) without asking
- Do not implement abstract-parsing (Source 3) until Fredrik explicitly asks for it
- Do not merge PubTator candidates into diseases.json without `--confirm` flag
